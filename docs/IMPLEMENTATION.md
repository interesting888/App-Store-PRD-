# 设计思路、实现过程与 AI 使用方式

## 1. 设计思路

这个项目的目标不是做一个静态报告页面，而是把“App Store 评论分析到产品方案”的流程产品化成一个可运行工具。设计时重点解决三个问题：

1. **数据可信**：所有评论必须来自在线采集、用户导入、缓存或明确标注的样例数据，不能编造。
2. **语义可泛化**：主题发现不能依赖某个 App 的固定关键词或预定义分类，需要能处理新 App、新目标和混合语言评论。
3. **结论可追溯**：从发现、PRD 到测试用例，每一步都必须能回到具体评论 ID 和摘录。

因此系统采用“确定性规则 + 运行时模型 + 证据校验”的混合架构：

- 规则负责采集、清洗、去重、字段规范化、统计和追溯验证。
- DeepSeek 负责动态主题发现这类开放域语义任务。
- 校验器负责拦截没有证据的模型输出，避免幻觉进入 PRD 和测试计划。

## 2. 整体架构

```text
React UI
  -> Express API
    -> URL 解析与范围确定
    -> 评论采集 / 导入 / 缓存 / 采集诊断
    -> 清洗、去重、统计
    -> DeepSeek 动态主题发现
    -> Finding 证据校验
    -> PRD 与版本规划生成
    -> 测试用例生成
    -> 追溯链验证
    -> 可靠性评估
    -> 评估标准核对
  -> UI 展示阶段进度和交付物
```

关键文件：

- `src/App.tsx`：前端工作台，展示输入区、阶段进度、方案说明和分析产物。
- `server/index.ts`：Express API，提供 `/api/health`、`/api/sample`、`/api/analyze`。
- `server/pipeline.ts`：核心流水线，串联采集、清洗、AI 分析、PRD、测试和校验。
- `shared/types.ts`：前后端共享的数据契约，保证交付物结构一致。
- `docs/AI.md`：模型配置、提示词和防幻觉策略。
- `docs/DATA_COLLECTION.md`：评论数据来源和局限说明。

## 3. 实现过程

### 3.1 输入和分析范围

用户输入 App Store 链接和分析目标。系统从链接中提取数字 App ID，并固定分析美国商店评论。

范围确定阶段使用规则解析：

- 是否关注低评分评论
- 是否限定版本
- 用户目标中是否出现订阅、转化、锻炼体验等提示

这些提示只用于约束样本范围，不作为最终主题分类依据。

### 3.2 评论采集和导入

采集策略分三层：

1. 优先尝试 Apple RSS 评论接口。
2. RSS 为空时尝试 App Store 页面结构化数据。
3. 如果在线数据不可用，则读取缓存或用户导入数据。

系统还支持 JSON / CSV 导入，方便用户用未见过的数据集测试泛化能力。兼容策略包括：

- 支持直接数组，也支持 `reviews`、`data`、`items`、`entries`、`records`、`results` 等常见容器。
- 支持 Apple RSS 风格的 `{ label: "..." }` 字段。
- 支持 `body`、`content`、`review`、`text`、`comment`、`description` 等评论正文字段别名。
- 支持 `rating`、`score`、`stars`、`im:rating` 等评分字段别名。
- CSV 解析支持 BOM、自动 trim 和宽松列数，减少外部数据格式差异带来的失败。

每次运行都会生成 `collectionReport`，用于解决“在线不可用时不知道为什么失败”的问题：

- 记录 RSS、网页结构化数据、缓存、样例或导入的每次尝试。
- 标注最终使用的数据来源。
- 明确列出采集局限，例如 RSS 空结果、网页结构化数据样本有限、缓存不一定最新。
- UI 的“可靠性”页会展示这些诊断信息。

### 3.3 清洗、去重和统计

清洗阶段完全使用规则，原因是这部分需要可复现：

- 统一字段格式
- 去除空正文和非法评分
- 按 review id 或文本 hash 去重
- 根据目标过滤评分或版本
- 计算评分分布、版本分布、样本量变化

清洗报告会展示输入数、输出数、重复数、非法数和过滤数。

### 3.4 动态主题发现

语义分析阶段调用 DeepSeek。模型输入包含：

- 用户分析目标
- 清洗后的评论样本
- 每条评论的 `id`、`rating`、`title`、`body`

模型输出必须是 JSON，包含：

- 主题标题
- 用户问题
- 严重度
- 置信度
- 支持评论 ID
- 冲突评论 ID
- 不确定性说明

服务端随后执行证据白名单校验：模型返回的 review id 必须存在于输入评论中。没有有效证据的发现会被删除。

### 3.5 PRD 和版本规划

PRD 生成以 Finding 为输入，按严重度和目标相关性拆分版本：

- 高严重度进入 `v1.0 修复版`
- 中等严重度进入 `v1.1 体验版`
- 低严重度或探索项进入 `v2.0 探索版`

每条需求包含：

- 问题陈述
- 需求描述
- 验收标准
- 不在范围
- 优先级
- 目标版本
- 来源 finding id
- 来源 review id
- 证据类型
- 不确定性说明

### 3.6 测试用例和追溯验证

每条需求生成两类测试用例：

1. 主流程验证
2. 回归与边界验证

追溯链为：

```text
Review -> Finding -> Requirement -> TestCase
```

验证器会检查：

- Finding 是否有真实评论证据
- Requirement 是否引用存在的 Finding 和 Review
- TestCase 是否引用存在的 Requirement
- TestCase 是否保留关联评论 ID

如果模型不可用，追溯验证会增加 warning，提醒正式评估需配置模型后重跑。

### 3.7 可靠性评估

为了让结果不只是“生成出来”，系统新增可靠性评估阶段。它会综合以下信号给出 0 到 100 的评分：

- 清洗后的样本量是否足够。
- 是否成功使用运行时模型完成动态主题发现。
- 是否存在追溯链错误或警告。
- 是否使用了离线样例数据。
- 是否处理了用户导入数据。

可靠性结果包含：

- `score`：综合评分。
- `level`：`high`、`medium`、`low`。
- `reasons`：支撑可信度的原因。
- `limitations`：数据局限和使用边界。
- `generalizationChecks`：无硬编码、导入兼容、运行时模型、证据链完整性的检查结果。

这个阶段用于回答评估中的关键问题：应用是否能在不进行特定于应用程序硬编码的情况下，对未见过的数据得出可靠结果。

### 3.8 评估标准核对

系统新增 `evaluation` 交付物和 UI 中的“评估核对”页，逐项对应产品质量标准：

1. 数据是否真实可靠、可复现，并说明来源和局限。
2. 清洗、分类和分析是否合理，是否揭示具体用户问题。
3. 模型驱动语义分析是否超越固定规则，并能推广到未见数据、应用和目标。
4. 是否区分证据、确定性统计、模型生成结论、不确定性和冲突反馈。
5. PRD 是否基于用户问题，并包含需求边界、优先级和版本规划。
6. 测试用例是否覆盖 PRD，并可追溯到相应用户评价。
7. UI 是否清晰展示流程和结果，项目是否可本地运行并有交付说明。

每一项核对都会输出：

- `status`：`pass`、`warning` 或 `fail`。
- `summary`：当前运行为什么满足或不满足该项。
- `evidence`：用于支撑判断的运行数据、统计或文档说明。
- `improvement`：当存在 warning 或 fail 时给出改进建议。

### 3.9 用户体验改进

界面新增了面向真实用户的交互能力：

- 分析目标快捷标签，减少用户输入成本。
- JSON / CSV 文件上传，支持用户直接提供数据集。
- 导入内容字符数提示，帮助用户判断当前是否在走在线采集还是导入分析。
- “评估核对”标签页逐项显示质量要求是否达标、证据和改进建议。
- “可靠性”标签页展示评分、局限和泛化能力检查。
- “方案说明”页结合当次运行状态解释设计思路和 AI 使用方式。

## 4. AI 的使用方式

### 4.1 模型配置

当前默认使用 DeepSeek：

```env
LLM_BASE_URL=https://api.deepseek.com/chat/completions
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-chat
LLM_TEMPERATURE=0.2
```

密钥通过 `.env` 中的 `LLM_API_KEY` 提供，不提交到仓库。

### 4.2 为什么选择模型做主题发现

评论中的问题表达通常是开放式的，例如：

- 用户可能用不同语言描述订阅困惑。
- 同一个问题可能出现多种措辞。
- 用户目标可能临时变化，例如只关注低评分、某个版本或锻炼体验。

固定关键词表很难覆盖这些变化，所以系统把“动态主题发现”和“问题整合”交给模型。

### 4.3 为什么不把所有事情都交给模型

数据清洗、去重、统计和追溯校验需要确定性和可审计。如果交给模型，结果会不稳定，也难以证明每条结论来自真实评论。

因此系统只让模型处理语义部分，最终是否采纳仍由规则校验决定。

### 4.4 防幻觉策略

系统使用多层防护：

- 低温度配置，减少随意生成。
- 提示词要求返回严格 JSON。
- 如果模型接口不接受 `response_format`，自动重试纯提示词 JSON 模式。
- 模型请求有超时保护，避免在线分析卡死。
- 每个 Finding 必须引用输入中的 review id。
- 服务端过滤不存在的 review id。
- 无有效证据的 Finding 会被删除。
- PRD 和测试用例继承 Finding 的 review id。
- 最后统一做追溯链验证。

## 5. 故障处理

系统显式处理以下失败：

- App Store RSS 返回空
- 页面结构化数据不可用
- 无外网
- 用户导入数据字段不完整
- DeepSeek API 未配置、超时或返回非 JSON
- 模型引用不存在的 review id

失败时系统不会生成虚假结论，而是：

- 记录错误或 warning
- 降级到缓存、导入数据或样例数据
- 模型失败时使用规则分桶结果
- 在 UI 中显示 `fallbackReason`
- 在可靠性和评估核对中明确标注未满足运行时 AI 或证据不足

## 6. 硬性要求映射

本项目把新增硬性要求显式落到代码和 UI：

- **不是纯爬虫或纯展示**：`Finding -> Requirement -> TestCase` 全链路生成，并在评估核对中检查。
- **从真实评论识别问题并转产品方案**：每个 Finding 带 evidence，每个 Requirement 带 sourceReviewIds，每个 TestCase 带 linkedReviewIds。
- **不只抓页面可见内容**：优先 RSS 或导入数据；网页结构化数据只是降级方案，并在 collectionReport 中说明。
- **PRD 可追溯**：追溯验证检查 Requirement 的 sourceReviewIds。
- **测试验证对应需求**：每个 TestCase 引用 requirementId 和 linkedReviewIds。
- **运行时 AI**：DeepSeek 动态主题发现是运行时调用；未调用成功会在评估核对中失败。
- **未见数据和故障场景**：JSON/CSV 兼容导入、去重、冲突证据、不确定性、采集失败和模型失败都会透明展示。
- **数据有限不伪造**：样本量、样例、缓存、网页结构化数据等局限都进入可靠性和评估核对。

## 7. 演示建议

建议按以下顺序演示：

1. 打开页面，说明输入 App Store 链接和分析目标。
2. 展示“方案说明”页，解释设计思路和 AI/规则分工。
3. 点击“加载离线样例”跑通无外网场景。
4. 用 JSON 或 CSV 粘贴新数据，证明没有特定 App 硬编码。
5. 展示 Findings 中的评论证据和冲突证据。
6. 展示 PRD 的来源 review id。
7. 展示测试用例如何关联需求和评论。
8. 展示追溯验证结果和完整 JSON。
